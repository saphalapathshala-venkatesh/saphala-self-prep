Implement a zero-cost Forgot Password flow in the Saphala Self Prep STUDENT FRONTEND APP using identity verification with the last 4 digits of the registered mobile number.

IMPORTANT
- This feature belongs in the STUDENT FRONTEND APP, not the admin app
- Admin app may keep its existing manual reset flow separately as a support backup
- Do not use email or SMS
- Keep implementation launch-safe, simple, and reliable
- Audit first before changing anything

GOAL
Allow a student to reset password by:
1. Clicking “Forgot password?” from login page
2. Entering email or mobile
3. Entering the last 4 digits of their registered mobile number
4. If verified, setting a new password
5. Revoke all existing sessions after successful reset
6. Redirect user back to login with a success message

PHASE 1 — AUDIT FIRST
Read and audit all relevant files before editing:
- prisma/schema.prisma
- User model
- Session model
- app/login page and LoginForm
- auth routes (login, register, logout, me)
- password hashing utilities
- session revoke logic
- any admin/manual password reset logic that can be reused safely
- any shared validation helpers

Before implementing, provide a short report:
- how student auth currently works
- where password hashing is done
- how sessions are revoked
- whether mobile is stored normalized
- whether email/mobile login uses identifier normalization
- safest place to add forgot-password flow

IMPLEMENTATION REQUIREMENTS

1) LOGIN PAGE
Add a clear “Forgot password?” link on the student login page.

2) NEW STUDENT PAGE
Create a student-facing page, for example:
- /forgot-password
This page should support the full reset flow.

Preferred UX:
Step 1:
- Identifier input: Email or Mobile
- Last 4 digits of registered mobile

Step 2 (only if verified):
- New password
- Confirm new password

3) VERIFICATION RULE
User must provide:
- valid identifier (email or mobile)
- correct last 4 digits of the mobile number stored for that user

Verification rules:
- use the same identifier normalization approach already used in login
- compare last 4 digits safely
- do not expose whether email/mobile exists separately from mismatch details more than necessary
- but keep UX practical for launch

Recommended user-facing error:
- “Details do not match our records.”

4) API ROUTES
Implement safe student-auth routes, for example:
- POST /api/auth/forgot-password/verify
- POST /api/auth/forgot-password/reset

OR a single safe route if simpler, but keep logic clean.

Verify route:
- accepts identifier + mobileLast4
- validates user exists and is active
- if valid, returns a short-lived verification token or safe server-side verification state

Reset route:
- accepts verification proof + newPassword + confirmPassword
- validates password rules
- updates passwordHash using bcryptjs
- revokes all active sessions for that user
- returns success

5) SECURITY / SAFETY
- Use bcryptjs only
- Never leak passwordHash
- Revoke all user sessions after password reset
- Do not log raw passwords
- Keep reset verification short-lived if tokenized
- Add basic validation and rate-limit style protections if easy and safe
- Keep implementation simple and robust

6) PASSWORD VALIDATION
Use existing password validation helper if available.
Otherwise enforce the app’s current password rules consistently.

7) SESSION HANDLING
After successful password reset:
- revoke/delete all active sessions for that user
- force fresh login

8) UI MESSAGES
Use clear student-friendly messages:
- verification mismatch: “Details do not match our records.”
- reset success: “Password reset successful. Please log in with your new password.”
- unexpected error: “Something went wrong. Please try again.”

9) OPTIONAL BUT GOOD
If mobile is missing for a user, show:
- “Password reset is not available for this account. Please contact support.”
Only do this if relevant in current schema/data.

10) TESTING
After implementation:
- verify valid identifier + correct last 4 works
- verify wrong last 4 fails
- verify password changes successfully
- verify old password stops working
- verify new password works
- verify all old sessions are revoked
- verify TypeScript/build passes
