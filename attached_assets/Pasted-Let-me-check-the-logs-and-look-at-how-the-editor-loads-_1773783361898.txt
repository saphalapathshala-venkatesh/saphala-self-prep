Here's exactly what happened and what's fixed:

Root cause — a bug in the EBook creation route

When you first created the EBook and saved it, the API's create handler had contentBlocks missing from the code that writes pages to the database:

// Bug: contentBlocks silently dropped on creation
create: incomingPages.map(p => ({
  title: p.title,
  contentHtml: p.contentHtml,  // ← contentBlocks never appeared here
  orderIndex: p.orderIndex,
}))
The block editor sent your tables, boxes, and paragraphs correctly — but the server quietly ignored and discarded them. The pages were created with contentBlocks = null in the database. When you re-opened the EBook, the editor loaded null → empty → nothing visible.

What's fixed

The create handler now correctly includes contentBlocks when saving each page. Going forward, creating an EBook with full content will persist everything properly.

Regarding your existing EBook

Unfortunately the content you typed was never written to the database — it only lived in the browser at the time. You will need to re-enter the content for those 8 pages. Once you add the blocks and click Save, it will use the edit route (which was already correct) and everything will be persisted and visible in both the admin editor and the student frontend.
