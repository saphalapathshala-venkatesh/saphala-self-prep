Root cause

The schema has a single canonical field for each identity (email String? @unique, mobile String? @unique). There are no secondary normalized fields. The admin PUT handler and student signup both already normalized email to lowercase and trimmed mobile before writing — so edits correctly freed the old value in the DB.

The actual bug was in the login routes, not in the edit flow itself:

# Stored in DB (by signup or admin PUT — both lowercase email):
email = "new@example.com"
# User tries to log in after admin edited their email:
identifier = "New@Example.com"   ← user remembers it with mixed case
# Login lookup (before fix):
WHERE email = 'New@Example.com'  ← PostgreSQL case-sensitive → NO MATCH → "Invalid credentials"
With login broken, the user couldn't access their account. When they tried to re-register using the NEW email (in any case), signup lowercased it to new@example.com, found the existing record, and returned "already exists" — trapping them completely.
