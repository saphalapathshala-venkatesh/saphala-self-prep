How to wire entitlement in your student frontend app
The simplest, cleanest pattern is a single API call per session that returns the student's set of entitlements, plus a check function at each content render:

1. Entitlement resolution endpoint (already exists: /api/users/[id]/entitlements)
On login, fetch the student's active entitlements and cache them (e.g. in React Context or a Zustand store). Each entitlement has a courseId it covers.

2. Course → content mapping
Each content item (Video, Test, FlashcardDeck, EBook, PDF) is linked to one or more Courses. The student's entitlement grants access to all content linked to that course.

3. The access gate — check at render time

function canAccess(content: { isFree: boolean; courseIds: string[] }, entitledCourseIds: Set<string>): boolean {
  if (content.isFree) return true;                          // free for everyone
  return content.courseIds.some(id => entitledCourseIds.has(id)); // paid — must own a linked course
}

For Video, use accessType === "FREE" instead of isFree.

4. Practical flow

Student opens a flashcard deck → frontend checks canAccess(deck, myEntitlements)
If false → show a "Purchase required" paywall modal pointing to the relevant course(s)
If true → load and render the content normally
5. Anti-spoofing
Always re-validate on the server in every content-serving API route (/api/student/decks/[id]/cards, etc.) — the frontend check is just UX, the server is the real gate.
