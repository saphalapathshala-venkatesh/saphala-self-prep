Feature 1 — Reports
Learner-wise reports
Status: Partially Exists

What exists: /admin/users/[id]/attempts — a read-only table showing every attempt for one user, with columns: test name, attempt #, status, score%, correct, wrong, unanswered, time used, XP. Both an admin page (server-rendered) and an API (GET /api/admin/users/[id]/attempts) exist.
What is missing: No subject breakdown per attempt in this view. No filters (date range, test type, status). No topic or question drill-down. No aggregate summary stats across attempts (e.g., avg score, improvement trend). No export.
Schema sufficient: Yes, for significant enrichment of this view.
Test-wise reports
Status: Not Present

What exists: getAllResultsForTest(testId) in resultStore.ts, but this is in-memory and used only for leaderboard computation — not exposed in admin, not persisted.
What is missing: No admin page or API that shows "all learners who attempted Test X" with their scores, timing, ranks.
Schema sufficient: Yes — Attempt table filtered by testId gives all needed data.
Section-wise reports
Status: Not Present

What exists: TestSection model and TestQuestion.sectionId exist in the schema.
What is missing: generate-result ignores sections entirely — only subject-wise breakdown is computed. No section score breakdown anywhere (admin or student).
Schema sufficient: Computable at query time by joining AttemptAnswer → TestQuestion.sectionId.
Topic/subtopic-wise reports
Status: Data Exists but UI Missing

What exists: Question.topicId and Question.subtopicId are in the schema. AttemptAnswer.questionId can be joined to Question.
What is missing: No compute path groups AttemptAnswers by topicId. No admin or student view shows topic-level accuracy.
Schema sufficient: Yes — multi-join query can produce this at query time.
Time-based reports (trends over time)
Status: Data Exists but UI Missing

What exists: Attempt.startedAt, Attempt.submittedAt, AttemptAnswer.timeSpentMs per question are all stored. The admin attempts table shows totalTimeUsedMs as a column.
What is missing: No trend charts, no time-series aggregation, no "attempts per month" or "avg time vs score" view.
Schema sufficient: Yes.
Downloadable/exportable reports
Status: Not Present

No CSV export, no PDF generation, no data export endpoint anywhere in the codebase.

Feature 3 — Compare Last 3 Tests
Status: Not Present

The per-test attempts page (/testhub/tests/[testId]/attempts) shows multiple attempts for one test in a list — but this is same-test only, not cross-test comparison.
The admin attempts table lists all attempts chronologically — no side-by-side, no delta, no visual trend.
Data sufficient? Yes — Attempt records for the same user across different tests all have scorePct, correctCount, totalTimeUsedMs, submittedAt. Comparing the last 3 is a straightforward query. Subject breakdown would require re-computing from AttemptAnswer per attempt (or storing it as persisted columns).
Feature 4 — Improvement / Deterioration Insights
Status: Not Present

No delta computation between attempts anywhere.
No alerts, no badge, no text like "Score improved by 12% from last attempt."
Data sufficient? Yes for overall score trend — Attempt.scorePct by attemptNumber or submittedAt is all that's needed. For subject-level improvement, the subject breakdown would need to be re-derived from AttemptAnswer (doable at query time, slightly expensive).
Feature 5 — Mistake Tracking
Status: Not Present

What exists: AttemptAnswer stores isCorrect = false and questionId for every wrong answer.
What is missing: No query groups repeated wrong answers by questionId across multiple attempts. No "mistake log" page in admin or student app. No "repeatedly wrong questions" detection. No weak-question-type analysis.
Data sufficient? Yes — a group-by query on AttemptAnswer WHERE isCorrect=false GROUP BY questionId gives a full mistake frequency table per learner. Joining to Question.topicId, Question.difficulty, Question.type gives weak-topic and weak-type patterns.
Feature 6 — Recommendation Engine / Guidance
Status: Not Present

No recommendation logic exists anywhere — admin or student side.
No "what to study next," no "try this test," no "revise this topic" feature.
Data sufficient for rule-based recommendations? Partially. The mistake tracking and weakness data can be derived from AttemptAnswer. Linking weakness topics to available ContentPage, FlashcardDeck, or future tests would require additional mapping (e.g., a topic → resource index). The taxonomy models (Subject/Topic/Subtopic) exist and ContentPage.subtopicId + FlashcardCard.subtopicId are stored, so a simple "you got these topics wrong → here are resources for those topics" rule is feasible with existing data. A proper ML-based recommendation engine is out of scope given current data volume.
