Two Issues Worth Knowing About
1. Flashcard deck preview hint uses browser local timezone — the small text below the Unlock At field reads "Students can access this deck from … onwards" but uses new Date(value).toLocaleString() without any timezone specification. On a machine set to IST this appears correct, but it does not enforce IST display the way every other product does.

2. Edit pages show the wrong time when re-opening a record — only the Tests page correctly converts existing unlock times back to IST for display in the input. All other products (Live Classes, Videos, E-books, PDFs, Flashcards) call .slice(0, 16) on the raw UTC ISO string returned by the API. For example, if you stored 10:30 AM IST, the edit form shows 5:00 AM in the input (the UTC equivalent). If an admin then saves without adjusting the time, the unlock shifts 5 hours 30 minutes earlier with each save.

Summary in One Line
The admin app treats all unlock times as IST (UTC+5:30) at the point of entry. Values are stored as UTC in the database. The +05:30 offset is appended as a hardcoded string during the API call. Tests is the only product type that correctly reconverts UTC → IST when populating the edit form; all others show the raw UTC time in the edit input.
