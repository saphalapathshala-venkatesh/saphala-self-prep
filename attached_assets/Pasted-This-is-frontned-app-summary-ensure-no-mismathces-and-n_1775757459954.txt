What was fixed and why:

Root cause: When an admin opens an edit form for a record that already has an unlockAt saved, the API returns the stored time as UTC (e.g., 05:00 UTC). The old code used .slice(0, 16) which placed that raw UTC string into the datetime-local input. When the admin saved — even without touching the time — the UI would append :00+05:30, treating 05:00 as IST and shifting the stored time to 23:30 UTC (5h30m earlier than the original intent). Every re-save compounded the drift.
