TASK — One-device-only login enforcement with transparent signup/login notice

Goal:
Implement one-device-only student access cleanly and transparently.

Business rule:
- By default, a student account can be used on only one device/browser at a time
- If the student tries to log in from another device/browser, login must be blocked
- Do NOT silently replace the existing device
- Show a clear warning message
- Keep the implementation understandable and user-friendly

Important context:
- Admin-side device fields and controls may already exist in the shared schema
- Frontend must enforce the rule properly through auth/session logic
- The checkbox on signup/login is only acknowledgment, not the enforcement itself

REQUIRED WORK

PART A — Audit existing device logic
1. Inspect current schema fields and auth logic related to:
   - maxWebDevices
   - allowMultiDevice
   - sessions
   - any stored device fingerprint / token / user-agent logic
   - admin reset-device logic if already present

2. Identify what already exists vs what is missing for proper one-device enforcement.

PART B — Implement actual enforcement
1. On first successful login:
   - bind the account to the current device/browser identity

2. On later login attempts:
   - if same device/browser → allow login
   - if different device/browser and user is restricted to one device → block login

3. Do not silently revoke the old device.
4. Do not allow second-device login by default.
5. Keep admin-reset compatibility intact.

Important:
- Use a clean and deterministic device identity strategy already compatible with the project where possible
- Do not implement a weak cosmetic-only frontend check
- Enforcement must happen in backend/auth logic, not just UI

PART C — Signup and login page transparency
Add a small colored informational card to both:
- signup page
- login page

Inside the card, show simple-language policy text about one-device-only usage.

Add a checkbox:
- label: “I understand the one-device policy.”

Behavior:
- user must check it before submitting signup/login
- style should be small, clean, visible, not aggressive
- no major page redesign
