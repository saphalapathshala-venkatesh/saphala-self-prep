PHASE 2 — FIX LOGIN ERROR MAPPING
Implement clear frontend mapping for backend auth error codes. The login form must NOT collapse all failures into one generic error.

Map these codes cleanly:
- INVALID_CREDENTIALS → “Incorrect email/mobile or password.”
- ACTIVE_SESSION_EXISTS → “This account is already logged in on another device. Please log out there or contact support.”
- ACCOUNT_BLOCKED → “Your account has been blocked. Please contact support.”
- ACCOUNT_INACTIVE → “Your account is inactive. Please contact support.”
- SESSION_EXPIRED → “Your session has expired. Please log in again.”
- fallback only for real unexpected errors → “Something went wrong. Please try again in a moment.”

UX RULES
- ACTIVE_SESSION_EXISTS should use an amber/yellow warning-style card, not a generic red error
- INVALID_CREDENTIALS should stay a normal red validation-style error
- Do not expose raw backend codes to the user
- Keep wording plain and student-friendly
