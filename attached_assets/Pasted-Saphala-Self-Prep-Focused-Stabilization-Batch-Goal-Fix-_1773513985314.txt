Locked business rules
A. Test state/button behavior
For a given student + test:
- No previous attempt exists -> show "Start Test"
- Existing in-progress resumable attempt -> show "Resume Test"
- At least one completed/submitted attempt exists and reattempts are allowed -> show "Reattempt"
- Max attempts reached -> disable CTA and show "Attempts Exhausted"
- Test is published but current time is before student-visible start time -> show "Starts on <date/time>"
- Test availability window passed / test unavailable -> show "Expired" or "Unavailable"

Important:
- After the first successfully submitted attempt, the student must NOT continue seeing "Start Test"
- They must see "Reattempt" if another attempt is allowed
- Use actual attempt state from DB, not only superficial UI state

B. XP rules by attempt number
For each test attempt:
- Attempt 1 -> 100% of original/base XP
- Attempt 2 -> 50% of original/base XP
- Attempt 3 and beyond -> 0 XP

This must be applied attempt-wise, not only test-wise.
Store enough data so admin can see:
- attempt number
- base XP
- awarded XP
- rule used / reason
- marks/score/percentage

C. Attempt history / result log
Each attempt must remain separate.
Do NOT merge attempts into one ambiguous result record.

For each attempt, preserve/show as available:
- attempt number
- startedAt
- submittedAt
- score / marks
- percentage
- time taken
- XP awarded
- status (in_progress / completed / auto_submitted / abandoned if such states exist)
- device/session metadata if already available or easy to add without destabilizing system

Admin should clearly see attempt-wise records.
Student side can remain simpler, but attempt separation must be preserved in backend logic and admin logs.

E. Save / answer persistence behavior

- Answer persists on Save & Next
- Answer persists when Marked for Review and answered
- Resume should restore saved state correctly
- Refresh/reconnect should not silently lose saved answers if attempt is resumable
- Reattempt must create a fresh attempt record and must not overwrite prior attempt answers

F. One-device login control
Current issue: user can login on two devices and attempt test across both. Fix this.

Expected behavior:
- By default, one learner account may be used on only one approved device/browser at a time
- If same learner tries login from another device/browser and they are NOT explicitly allowed multi-device access, block login
- Recommended block message: account already active on another device; contact support/admin or logout from previous device
- Admin override must continue to work if already designed:
  - allow specific learner multi-device access
  - reset learner device binding
  - global reset if such control already exists

Implementation guidance:
- Audit current session/device restriction logic end-to-end
- Find why second device is currently allowed
- Do not rely only on userId
- Use current session/device token design if already present
- If missing, implement the smallest robust mechanism possible:
  - stable device token/fingerprint stored securely
  - compare current device to allowed device binding
  - block mismatch unless admin override
- Avoid fragile IP-only logic
- Enforce this both at login/session creation level and at test-start/test-resume access level where relevant

G. Concurrent test protection
Very important:
- A learner should not be able to actively run the same test attempt from two devices/sessions simultaneously
- If a second device/session tries to access same active attempt, block it
- Only one active session/attempt context for a learner+attempt should be effective
- Prevent fairness/security loopholes

Expected behavior:

- Audit for mismatch between:
  - email vs identifier vs mobile
  - normalized vs raw values
  - bcrypt/bcryptjs compatibility
  - stale env/DB mismatches
  - session/token stale behavior
